# feat: реализовать интерактивный слайдер «Было/Стало» на главной странице

**Labels:** `ready-for-agent`

---

## Problem Statement

Посетитель главной страницы видит раздел «Живой пример» с фотографиями котика до и после перехода на питание Cat Energy Slim. HTML-разметка слайдера уже присутствует — кнопки «Было» и «Стало», ползунок (`slider__flag`), шкала (`slider__scale`) — но ни одна из этих интерактивных возможностей не работает. Кнопки не реагируют на клик, ползунок не двигается. Пользователь не может сравнить фото и оценить результат продукта.

## Solution

Реализовать JavaScript-логику слайдера в составе существующей сборки. Слайдер должен поддерживать два режима: переключение кнопками («Было» / «Стало») и перетаскивание флага-ползунка для плавного сравнения. Фотографии «до» и «после» разделяются вертикальной линией, положение которой определяется позицией флага.

## User Stories

1. Как посетитель главной страницы, я хочу кликнуть кнопку «Было», чтобы увидеть фото кота до начала программы питания.
2. Как посетитель главной страницы, я хочу кликнуть кнопку «Стало», чтобы увидеть фото кота после программы питания.
3. Как посетитель главной страницы, я хочу перетащить флаг-ползунок влево или вправо, чтобы плавно сравнить фото «до» и «после» в любой пропорции.
4. Как мобильный пользователь, я хочу перетаскивать флаг-ползунок касанием пальца (touch), чтобы сравнивать фото на смартфоне.
5. Как пользователь, я хочу видеть визуальное выделение активной кнопки («Было» или «Стало»), чтобы понимать, какое состояние сейчас отображается.
6. Как пользователь, я хочу, чтобы слайдер при загрузке страницы был установлен в положение «Было» (крайнее левое), чтобы сразу видеть исходное состояние.
7. Как пользователь, я хочу, чтобы при переключении кнопкой «Было» флаг перемещался в крайнее левое положение, чтобы отображение соответствовало нажатой кнопке.
8. Как пользователь, я хочу, чтобы при переключении кнопкой «Стало» флаг перемещался в крайнее правое положение, чтобы отображение соответствовало нажатой кнопке.
9. Как пользователь клавиатуры, я хочу использовать стрелочные клавиши для управления ползунком, когда он в фокусе, чтобы взаимодействовать со слайдером без мыши.
10. Как пользователь вспомогательной технологии, я хочу, чтобы кнопки «Было» и «Стало» имели описательные `aria`-атрибуты, чтобы понимать их назначение через скринридер.
11. Как посетитель десктопа, я хочу видеть плавную анимацию движения флага при клике на кнопку, чтобы переход был наглядным.
12. Как посетитель, я хочу, чтобы перетаскивание флага не выходило за пределы контейнера слайдера, чтобы не возникало визуальных артефактов.
13. Как посетитель, я хочу, чтобы слайдер корректно работал при изменении размера окна браузера (ресайз), чтобы интерактивность не ломалась на адаптивных брейкпоинтах.

## Implementation Decisions

- **Модуль слайдера** добавляется в `scripts.js` — дополнительный входной файл не нужен, сборка не меняется.
- **Привязка к DOM:** слайдер инициализируется по наличию элемента `.slider` на странице; если элемент не найден, инициализация молча пропускается (страницы `catalog.html` и `form.html` слайдера не имеют).
- **Режим кнопок:** клик по `.slider___toggle--before` устанавливает флаг в позицию 0 %; клик по `.slider___toggle--after` — в 100 %. Активная кнопка получает модификатор `slider___toggle--active`.
- **Режим перетаскивания:** слушатели `pointerdown` / `pointermove` / `pointerup` на `.slider__flag` обновляют `left`-позицию флага в пределах `[0%, 100%]` контейнера `.slider__range`. Элемент `.slider__scale` растягивается пропорционально, обеспечивая визуальный разделитель.
- **Видимость фото:** при перетаскивании оба элемента `slider__item` видны одновременно с clip-based разделением через позицию флага; при нажатии кнопки только соответствующий `slider__item` виден полностью.
- **Состояние при загрузке:** начальная позиция — «Было» (флаг крайний левый, кнопка «Было» активна).
- **GSAP:** анимация перехода флага при клике на кнопку реализуется через уже подключённый `window.gsap` (файл `gsap.min.js` уже загружается на `index.html`). Паттерн `window.gsap` уже применяется в `cat3d.js`.
- **Ресайз:** при `resize` пересчитываем границы перетаскивания.

## Testing Decisions

Тестирование ручное, без автоматизированного фреймворка.

Хороший тест проверяет **внешнее поведение**, видимое пользователю, а не детали реализации (не имена переменных, не вызовы GSAP напрямую).

Ручной чеклист для приёмки:

- [ ] Открыть `build/index.html` в браузере. Кнопка «Было» визуально активна, видно фото «до».
- [ ] Кликнуть «Стало» → видно фото «после», кнопка «Стало» активна, «Было» — нет.
- [ ] Кликнуть «Было» → обратно к фото «до».
- [ ] Перетащить флаг мышью вправо → отображение меняется плавно.
- [ ] Перетащить флаг за пределы контейнера → флаг остаётся внутри.
- [ ] Проверить на мобильной ширине (320px) через DevTools → тач-перетаскивание работает.
- [ ] Изменить размер окна → слайдер не ломается.
- [ ] Открыть `catalog.html` → слайдера нет, ошибок в консоли нет.

Аналогов тестов в кодовой базе нет — проект без тестового фреймворка.

## Out of Scope

- Добавление новых фотографий или изменение контента раздела «Живой пример».
- Реализация слайдера на страницах `catalog.html` и `form.html`.
- Сохранение позиции слайдера между визитами (localStorage).
- Автоматическое воспроизведение / анимация слайдера без действия пользователя.
- Полноэкранный просмотр фотографий.
- Поддержка браузеров без Pointer Events API (IE11 и ниже).

## Further Notes

В `index.html` уже подключён `gsap.min.js` глобально (`window.gsap`), что позволяет использовать GSAP для анимации флага без изменения сборки.

Разметка слайдера использует тройное подчёркивание в именах классов (`slider___controls`, `slider___toggle`) — это отклонение от БЭМ-конвенции, принятой в остальном проекте. Реализацию следует ориентировать на существующие классы, не переименовывая их.
